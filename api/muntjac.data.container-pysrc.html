 24      """A specialized set of identified Items. Basically the IContainer is a set 
 25      of L{Item}s, but it imposes certain constraints on its contents. These 
 26      constraints state the following: 
 27   
 28        - All Items in the IContainer must have the same number of Properties. 
 29        - All Items in the IContainer must have the same Property ID's (see 
 30          L{Item.getItemPropertyIds}). 
 31        - All Properties in the Items corresponding to the same Property ID must 
 32          have the same data type. 
 33        - All Items within a container are uniquely identified by their non-null 
 34          IDs. 
 35   
 36      The IContainer can be visualized as a representation of a relational 
 37      database table. Each Item in the IContainer represents a row in the table, 
 38      and all cells in a column (identified by a Property ID) have the same data 
 39      type. Note that as with the cells in a database table, no Property in a 
 40      IContainer may be empty, though they may contain C{None} values. 
 41   
 42      Note that though uniquely identified, the Items in a IContainer are not 
 43      necessarily L{IOrdered} or L{IIndexed indexed}. 
 44   
 45      Containers can derive Item ID's from the item properties or use other, 
 46      container specific or user specified identifiers. 
 47   
 48      If a container is L{filtered<IFilterable>} or L{sorted<ISortable>}, 
 49      most of the the methods of the container interface and its subinterfaces 
 50      (container size, L{containsId}, iteration and indices etc.) 
 51      relate to the filtered and sorted view, not to the full container contents. 
 52      See individual method doc-strings for exceptions to this (adding and 
 53      removing items). 
 54   
 55      The IContainer interface is split to several subinterfaces so that a class 
 56      can implement only the ones it needs. 
 57   
 58      @author: Vaadin Ltd. 
 59      @version: 1.0.2 
 60      """
